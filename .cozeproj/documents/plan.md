# 资源库文件类型支持与上传修复计划

## 概述

完善资源库对各种文件类型的支持能力，确保压缩文件、可执行安装包（.exe/.msi/.dmg/.deb/.rpm/.apk）、源码文件等都能正常上传和下载。同时修复大文件（>50MB）上传仍走 TUS 路径导致 413 错误的问题。

## 技术方案

| 维度 | 选择 | 理由 |
|------|------|------|
| 文件类型限制 | 无限制（全开放） | 资源库需要支持任意软件/代码资源，不做 MIME 白名单和扩展名过滤 |
| 大文件上传 | S3 对象存储分片上传（>50MB） | Supabase Storage TUS 端点有 50MB 硬限制，S3 对象存储无此限制 |
| 小文件上传 | Supabase Storage TUS 直传（≤50MB） | 小文件直接走 TUS，体验更好 |
| 下载 | 按 storage_type 生成对应签名 URL | 兼容两种存储后端 |

## 根因分析

当前代码已实现双存储策略（≤50MB TUS，>50MB S3 分片），但存在以下问题：

1. **生产环境未部署最新 S3 分流代码**：用户截图显示 121MB 文件仍请求 Supabase TUS 端点，返回 413
2. **文件类型支持已开放**：代码层面无 accept 限制、无扩展名过滤，但需要验证端到端流程
3. **下载时文件名保留**：需要确保下载时浏览器能正确识别文件类型

## 功能模块

### 1. 文件上传类型开放

- 前端 `<input type="file">` 无 `accept` 属性限制
- S3 分片上传和 TUS 路径均无扩展名/MIME 白名单
- `.exe`, `.zip`, `.rar`, `.7z`, `.tar`, `.gz`, `.deb`, `.rpm`, `.dmg`, `.msi`, `.apk`, `.iso` 等全支持

### 2. 大文件 S3 分片上传（已实现，需验证）

- `/api/upload/s3-chunk-init` — 初始化上传会话
- `/api/upload/s3-chunk` — 接收分片写入 /tmp
- `/api/upload/s3-chunk-complete` — 合并后通过 `streamUploadFile` 上传 S3

### 3. 下载体验优化

- 确保下载时 Content-Type 正确（根据文件扩展名推断）
- 确保下载时保留原始文件名

## 是否有原型设计

否 — 资源库页面已存在，本次为后端逻辑验证与完善，不涉及页面/UI 改动。

## 实施步骤

1. **验证上传代码文件类型支持完整性** — 检查 `resources/page.tsx`、`s3-chunk-init`、`tus-params` 等文件，确认无文件类型限制
2. **验证下载路由的文件名和 Content-Type 处理** — 检查 `download/route.ts`，确保各种扩展名能正确下载
3. **运行代码静态检查和接口测试** — `pnpm lint` + `pnpm ts-check` + 接口冒烟测试
4. **日志健康检查与交付** — 确认无异常后交付
